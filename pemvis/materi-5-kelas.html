Pertemuan 5: membangun pemahaman UI freeze, solusi QThread, lalu mengintegrasikan REST API ke aplikasi PySide6

0. Strategi Masuk Kelas — Threading ⏱ 10 menit

Catatan Dosen: Jangan mulai dari definisi thread. Mulai dari pengalaman yang sudah pernah mahasiswa rasakan: aplikasi yang "not responding" atau loading bar yang macet. Dari situ arahkan ke pertanyaan: kenapa itu bisa terjadi, dan bagaimana cara mencegahnya?

Skenario Pembuka

Tulis pertanyaan ini di papan: "Pernahkah kalian klik tombol di aplikasi, lalu aplikasinya diam dan tidak bisa diklik sama sekali selama beberapa detik?"

Setelah mahasiswa menjawab, lanjutkan: "Itu namanya UI freeze. Hari ini kita akan tahu kenapa itu terjadi dan cara memperbaikinya."

Alur Mengajar Threading yang Disarankan
  1. Tunjukkan bad.py — biarkan mahasiswa merasakan freeze sendiri.
  2. Tanya: "Kenapa tombol lain tidak bisa diklik? Kenapa data tidak muncul satu per satu?"
  3. Jelaskan konsep main thread dan kenapa ia tidak boleh diblokir.
  4. Tunjukkan compare.py — dua tab, biarkan mahasiswa membandingkan langsung.
  5. Baru jelaskan kode good.py — QThread, Signal, run().

1. Demo UI Freeze — bad.py ⏱ 15 menit

Strategi: Jalankan bad.py di depan kelas. Jangan jelaskan dulu — biarkan mahasiswa mengamati sendiri. Setelah klik "Mulai Proses", minta mereka klik tombol counter. Reaksi mereka saat counter tidak bertambah adalah momen pengajaran terbaik.

✅ Kode ini bisa di-copy langsung

bad.py BAD — Jalankan Dulu 
import sys
import time
from PySide6.QtWidgets import (
    QApplication, QWidget, QVBoxLayout,
    QPushButton, QLabel, QListWidget
)

class FreezeDemo(QWidget):
    def __init__(self):
        super().__init__()
        self.setWindowTitle("UI Freeze Demo")
        layout = QVBoxLayout(self)

        self.status = QLabel("Status: menunggu...")
        layout.addWidget(self.status)

        self.list_widget = QListWidget()
        layout.addWidget(self.list_widget)

        btn_proses = QPushButton("Mulai Proses")
        btn_proses.clicked.connect(self.proses_berat)
        layout.addWidget(btn_proses)

        # Tombol ini TIDAK bisa diklik saat proses_berat() berjalan
        self.counter = 0
        self.btn_test = QPushButton("Klik saya! (0)")
        self.btn_test.clicked.connect(self.tambah_counter)
        layout.addWidget(self.btn_test)

    def tambah_counter(self):
        self.counter += 1
        self.btn_test.setText(f"Klik saya! ({self.counter})")

    def proses_berat(self):
        self.list_widget.clear()
        self.status.setText("Status: memproses...")
        # Loop ini memblokir main thread — UI freeze total
        # Data tidak muncul satu per satu, melainkan sekaligus di akhir
        for item in ["Data A", "Data B", "Data C", "Data D", "Data E"]:
            time.sleep(1)
            self.list_widget.addItem(item)
        self.status.setText("Status: selesai")

if __name__ == "__main__":
    app = QApplication(sys.argv)
    w = FreezeDemo()
    w.show()
    sys.exit(app.exec())

Pertanyaan Pemantik Setelah Demo

  1. "Kenapa counter tidak bertambah saat proses berjalan?"
  2. "Kenapa data tidak muncul satu per satu padahal kode sudah memanggil addItem() di setiap iterasi?"
  3. "Apa yang sebenarnya sedang dilakukan main thread selama 5 detik itu?"
Penjelasan yang Perlu Disampaikan

Qt menggambar ulang UI dan memproses event (klik, ketik) melalui event loop yang berjalan di main thread. Selama proses_berat() berjalan, main thread sibuk — event loop tidak bisa berjalan, sehingga:

  • Klik tombol tidak diproses (counter tidak bertambah)
  • Perubahan widget tidak di-render (data tidak muncul bertahap)
  • Semua perubahan baru tampil sekaligus setelah fungsi selesai
Istilah Penjelasan untuk Kelas
Main Thread Thread utama tempat UI berjalan. Menggambar widget dan memproses klik user.
Event Loop Mesin di dalam Qt yang terus berjalan menunggu dan memproses event. Berhenti jika main thread diblokir.
UI Freeze Kondisi saat main thread diblokir oleh proses berat sehingga event loop tidak bisa berjalan.
Worker Thread Thread tambahan untuk menjalankan proses berat di latar belakang agar main thread tetap bebas.

Strategi: Sebelum menjelaskan kode, jalankan compare.py dulu. Biarkan mahasiswa sendiri yang merasakan perbedaannya di dua tab. Setelah mereka yakin ada perbedaan, baru buka kode dan jelaskan mengapa.

import sys
import time
from PySide6.QtCore import QThread, Signal
from PySide6.QtWidgets import (
    QApplication, QMainWindow, QWidget, QVBoxLayout,
    QPushButton, QLabel, QListWidget, QTabWidget
)

class WorkerThread(QThread):
    item_ready = Signal(str)
    finished   = Signal()

    def run(self):
        for item in ["Data A", "Data B", "Data C", "Data D", "Data E"]:
            time.sleep(1)
            self.item_ready.emit(item)
        self.finished.emit()

Cara Menjelaskan: Bandingkan bad.py dan good.py baris per baris. Tekankan bahwa struktur widget-nya identik — yang berbeda hanya di mana proses berat diletakkan dan bagaimana hasilnya dikirim balik ke UI.

good.py GOOD — Jelaskan Baris per Baris 
import sys
import time
from PySide6.QtCore import QThread, Signal
from PySide6.QtWidgets import (
    QApplication, QWidget, QVBoxLayout,
    QPushButton, QLabel, QListWidget
)

class WorkerThread(QThread):
    # Signal: satu-satunya cara aman kirim data dari worker ke main thread
    item_ready = Signal(str)
    finished   = Signal()

    def run(self):
        # run() dieksekusi di thread terpisah — main thread tetap bebas
        for item in ["Data A", "Data B", "Data C", "Data D", "Data E"]:
            time.sleep(1)
            self.item_ready.emit(item)   # kirim item ke UI
        self.finished.emit()

class GoodDemo(QWidget):
    def __init__(self):
        super().__init__()
        self.setWindowTitle("No Freeze Demo")
        layout = QVBoxLayout(self)
        self.status = QLabel("Status: menunggu...")
        layout.addWidget(self.status)
        self.list_widget = QListWidget()
        layout.addWidget(self.list_widget)
        self.btn_start = QPushButton("Mulai Proses")
        self.btn_start.clicked.connect(self.mulai_proses)
        layout.addWidget(self.btn_start)
        self.counter = 0
        self.btn_test = QPushButton("Klik saya! (0)")
        self.btn_test.clicked.connect(self.tambah_counter)
        layout.addWidget(self.btn_test)
        self.worker = None

    def tambah_counter(self):
        self.counter += 1
        self.btn_test.setText(f"Klik saya! ({self.counter})")

    def mulai_proses(self):
        self.list_widget.clear()
        self.status.setText("Status: memproses...")
        self.btn_start.setEnabled(False)
        # Buat worker, hubungkan signal, lalu start
        self.worker = WorkerThread()
        self.worker.item_ready.connect(self.tambah_item)
        self.worker.finished.connect(self.proses_selesai)
        self.worker.start()

    def tambah_item(self, item):
        self.list_widget.addItem(item)

    def proses_selesai(self):
        self.status.setText("Status: selesai")
        self.btn_start.setEnabled(True)

if __name__ == "__main__":
    app = QApplication(sys.argv)
    w = GoodDemo()
    w.show()
    sys.exit(app.exec())
Tiga Poin Wajib yang Harus Dipahami Mahasiswa
  1. Proses berat masuk ke run() — method ini dieksekusi di thread terpisah, bukan di main thread.
  2. Komunikasi via Signal — worker tidak boleh langsung mengubah widget. Kirim data lewat emit(), Qt yang mengurus pengirimannya ke main thread secara aman.
  3. Simpan referensi worker — self.worker = WorkerThread() wajib pakai self agar tidak di-garbage collect sebelum selesai.

3. Poin Diskusi Threading ⏱ 10 menit

Topik Yang Perlu Ditegaskan
Kenapa tidak update UI langsung dari run()? Qt tidak thread-safe untuk operasi UI. Update widget dari thread lain bisa menyebabkan crash atau perilaku tidak terduga.
Kenapa self.worker, bukan worker saja? Variabel lokal akan di-garbage collect setelah fungsi selesai. Thread yang masih berjalan akan crash. Simpan sebagai atribut instance.
Kapan perlu thread? Setiap kali ada operasi yang butuh waktu: network request, baca file besar, kalkulasi berat. Aturan: jika lebih dari ~100ms, pindah ke thread.
Signal vs langsung panggil method Signal aman lintas thread karena Qt mengantri pesan dan mengeksekusinya di thread yang tepat melalui event loop.
Kalimat yang bagus untuk ditekankan: "Aturan satu kalimat untuk threading di PySide6: proses berat di thread lain, update UI selalu di main thread lewat Signal."

Bug Umum yang Sering Dibuat Mahasiswa

  • Langsung memanggil self.list_widget.addItem() dari dalam run() — harusnya lewat Signal.
  • Tidak menyimpan worker sebagai self.worker — thread berhenti tiba-tiba.
  • Membuat thread baru setiap klik tanpa mengecek apakah thread sebelumnya masih berjalan — bisa tumpang tindih.
Transisi ke REST API

Setelah mahasiswa paham threading, sampaikan: "Sekarang kita akan pakai konsep yang sama untuk mengambil data dari internet. Network request adalah salah satu proses berat yang paling sering menyebabkan UI freeze — dan itulah yang akan kita pelajari selanjutnya."

4. Strategi Masuk — REST API ⏱ 10 menit

Catatan Dosen: Jangan mulai dari definisi REST API. Mulai dari kebutuhan nyata: "Kalau aplikasi desktop mau ambil data berita, cuaca, atau daftar mahasiswa dari internet, caranya bagaimana?" Hubungkan juga dengan threading yang baru dipelajari: network request adalah contoh nyata proses berat yang harus dijalankan di thread terpisah.

Skenario Pembuka

Tulis pertanyaan ini di papan: "Kalau Python bisa ambil data dari server, kenapa hasilnya jangan berhenti di terminal?" Dari sini arahkan bahwa output program seharusnya bisa dipindahkan ke GUI.

Alur Mengajar yang Disarankan
  1. Tunjukkan request sederhana ke server.
  2. Tunjukkan hasil JSON di terminal.
  3. Tanya: "Apakah user akhir mau membaca data di CLI?"
  4. Pindahkan output ke widget PySide.
  5. Naikkan level dari /posts/1 ke /posts.

5. Menjelaskan REST API ⏱ 15 menit

Cara Menjelaskan: Pakai analogi restoran. Client adalah pembeli, server adalah dapur, endpoint adalah meja layanan, dan JSON adalah isi pesanan yang dikirim balik.
Istilah Bahasa Sederhana untuk Kelas
Client Aplikasi kita yang meminta data
Server Tempat data disimpan dan dilayani
Endpoint Alamat data, misalnya /posts atau /users
GET Minta atau mengambil data
JSON Format data yang mudah dibaca program
Kalimat Ringkas untuk Disampaikan

REST API adalah cara aplikasi kita berbicara dengan server menggunakan HTTP. Untuk materi awal, cukup tekankan bahwa GET dipakai untuk mengambil data, lalu data itu kita olah dan tampilkan ke antarmuka.

7. Dosen Mengarahkan: Dari print() ke PySide6 ⏱ 20 menit

Strategi: Pada tahap ini, jangan langsung pakai tabel. Cukup pakai QLabel atau QTextEdit agar mahasiswa melihat perpindahan konsep dari output terminal ke output GUI.

✅ Kode ini bisa di-copy langsung

api_satu_medicine_gui.py Live Coding 
import sys
import requests
from PySide6.QtWidgets import QApplication, QWidget, QVBoxLayout, QPushButton, QTextEdit

BASE_URL = "https://api.pahrul.my.id/api"

class MedicineViewer(QWidget):
    def __init__(self):
        super().__init__()
        self.setWindowTitle("Viewer Satu Medicine")
        self.resize(500, 300)

        layout = QVBoxLayout(self)

        self.output = QTextEdit()
        self.output.setReadOnly(True)

        self.btn_load = QPushButton("Ambil Medicine 1")
        self.btn_load.clicked.connect(self.load_medicine)

        layout.addWidget(self.btn_load)
        layout.addWidget(self.output)

    def load_medicine(self):
        response = requests.get(f"{BASE_URL}/medicines/1")
        result = response.json()
        medicine = result["data"]  # data ada di dalam key 'data'

        teks = (
            f"ID: {medicine['id']}\n"
            f"Name: {medicine['name']}\n"
            f"Brand: {medicine['brand']}\n\n"
            f"Manufacturer:\n{medicine['manufacturer']}"
        )
        self.output.setPlainText(teks)

app = QApplication(sys.argv)
window = MedicineViewer()
window.show()
sys.exit(app.exec())
Poin Mengajar yang Harus Ditekankan
  • Data tetap diambil dengan requests.get().
  • Perbedaannya hanya di output: bukan print(), tetapi setPlainText().
  • Response API ini membungkus data di dalam key "data" — mahasiswa harus paham cara mengaksesnya.
  • Mahasiswa harus paham bahwa API dan GUI adalah dua lapisan yang berbeda.

Mini Tantangan Saat Live Coding

  1. Minta mahasiswa ganti endpoint ke /medicines/2.
  2. Minta mereka tampilkan hanya nama medicine di widget.
  3. Minta mereka tambahkan label "Data berhasil dimuat".

8. Naik Level - Menampilkan Data Lebih dari Satu ⏱ 25 menit

Inilah bagian penting: setelah mahasiswa melihat satu objek, arahkan ke endpoint /medicines. Jelaskan bahwa sekarang response.json()["data"] menghasilkan list berisi banyak dictionary.

import sys
import requests
from PySide6.QtWidgets import QApplication, QWidget, QVBoxLayout, QPushButton, QListWidget

  • Kalau datanya banyak, maka harus di-loop.
  • Widget yang cocok: QListWidget, QTableWidget, atau QTextEdit.

9. Poin Diskusi REST API di Kelas ⏱ 10 menit

Topik Yang Perlu Ditegaskan

import sys

from PySide6.QtWidgets import (
    QApplication, QWidget, QVBoxLayout, QPushButton,

class MedicineTable(QWidget):
    def __init__(self):
        super().__init__()
        self.setWindowTitle("Tabel Medicines")
        self.resize(700, 400)

        layout = QVBoxLayout(self)
        self.btn_load = QPushButton("Load Medicines")
        self.table = QTableWidget()
        self.table.setColumnCount(3)
        self.table.setHorizontalHeaderLabels(["Nama", "Brand", "Manufacturer"])

        self.btn_load.clicked.connect(self.load_medicines)

        layout.addWidget(self.btn_load)
        layout.addWidget(self.table)

    def load_medicines(self):
        response = requests.get(f"{BASE_URL}/medicines")
        result = response.json()
        medicines = result["data"]

        self.table.setRowCount(len(medicines))

        for row, medicine in enumerate(medicines):
            self.table.setItem(row, 0, QTableWidgetItem(medicine["name"]))
            self.table.setItem(row, 1, QTableWidgetItem(medicine["brand"]))
            self.table.setItem(row, 2, QTableWidgetItem(medicine["manufacturer"]))

sys.exit(app.exec())

Catatan Penting untuk Dosen

Untuk pengantar REST API, request langsung di method tombol masih bisa diterima. Tetapi beri catatan bahwa pada aplikasi yang lebih serius, request sebaiknya dipindah ke thread agar UI tidak freeze.

12. Penutup Kelas ⏱ 10 menit

Penutup yang Disarankan

Minta 2-3 mahasiswa mendemokan hasil GUI mereka. Pilih yang berbeda: satu yang menampilkan satu medicine, satu yang menampilkan list medicines, dan satu yang memakai tabel dengan QTableWidget.

Kalimat Penutup untuk Mahasiswa

Hari ini kalian belajar dua hal yang saling berkaitan: pertama, kenapa UI bisa freeze dan bagaimana QThread menyelesaikannya. Kedua, bagaimana mengambil data dari internet dan menampilkannya ke aplikasi desktop. Keduanya akan digabungkan di materi berikutnya — network request yang berjalan di thread terpisah agar UI tetap responsif.
