1. UI Freeze Problem

Dalam aplikasi GUI, seluruh tampilan dijalankan di Main Thread (UI Thread). Jika kita menjalankan proses berat (seperti network request, baca file besar, kalkulasi kompleks) di thread yang sama, maka UI akan freeze/hang sampai proses selesai.

Golden Rule of Threading
JANGAN pernah jalankan
proses berat di Main Thread
Tanpa Threading (FREEZE)
  • API call di main thread
  • UI membeku 2-10 detik
  • Tombol tidak bisa diklik
  • "Not Responding" muncul
  • User experience buruk
Dengan Threading (RESPONSIF)
  • API call di worker thread
  • UI tetap responsif
  • Loading indicator aktif
  • User bisa cancel request
  • Professional experience

2. QThread + Worker Pattern

Apa itu Thread?

Secara default, sebuah program berjalan dalam satu alur eksekusi yang disebut thread. Thread adalah unit terkecil dari eksekusi yang dikelola oleh sistem operasi. Bayangkan thread seperti seorang pekerja yang mengerjakan tugas satu per satu secara berurutan.

Single Thread vs Multi Thread
Single Thread
Tugas A → B → C
(antri, satu per satu)
vs
Multi Thread
Thread 1: Tugas A
Thread 2: Tugas B (bersamaan)

Dengan multi-threading, kita bisa menjalankan beberapa tugas secara bersamaan (concurrent). Di PySide6, Main Thread bertugas menggambar UI dan merespons input user. Tugas berat seperti network request harus dipindahkan ke Worker Thread agar Main Thread tetap bebas.

Istilah Penjelasan
Main Thread Thread utama tempat UI berjalan. Semua update widget HARUS dari sini.
Worker Thread Thread tambahan untuk menjalankan proses berat di latar belakang.
Concurrency Beberapa tugas berjalan "seolah bersamaan" (bisa bergantian cepat).
Parallelism Beberapa tugas benar-benar berjalan bersamaan di CPU berbeda.

import sys
import time
from PySide6.QtCore import QThread, Signal
from PySide6.QtWidgets import (
    QApplication, QWidget, QVBoxLayout,
    QPushButton, QLabel, QListWidget
)


# ─── Worker: class yang mewarisi QThread ───────────────────────────────────
# Semua proses berat diletakkan di dalam method run()
# Method run() dieksekusi di thread terpisah, BUKAN di main thread
class WorkerThread(QThread):

    # Signal untuk mengirim data ke UI — satu-satunya cara aman
    # berkomunikasi dari worker thread ke main thread
    item_ready = Signal(str)   # dikirim setiap kali satu item selesai diproses
    finished   = Signal()       # dikirim saat semua item selesai

    def run(self):
        # Ini berjalan di thread terpisah — main thread tetap bebas
        data = ["Data A", "Data B", "Data C", "Data D", "Data E"]
        for item in data:
            time.sleep(1)
            self.item_ready.emit(item)   # kirim item ke UI — tidak freeze!
        self.finished.emit()


class GoodDemo(QWidget):
    def __init__(self):
        super().__init__()
        self.setWindowTitle("No Freeze Demo")
        layout = QVBoxLayout(self)

        self.label = QLabel("Klik tombol untuk mulai proses")
        layout.addWidget(self.label)

        self.list_widget = QListWidget()
        layout.addWidget(self.list_widget)

        self.btn_start = QPushButton("Mulai Proses (tidak freeze!)")
        self.btn_start.clicked.connect(self.mulai_proses)
        layout.addWidget(self.btn_start)

        # Tombol ini tetap bisa diklik saat proses berjalan di background
        btn_test = QPushButton("Cek Responsif")
        btn_test.clicked.connect(lambda: self.label.setText("UI tetap responsif!"))
        layout.addWidget(btn_test)

        self.worker = None

    def mulai_proses(self):
        self.list_widget.clear()
        self.label.setText("Memproses...")
        self.btn_start.setEnabled(False)

        self.worker = WorkerThread()
        self.worker.item_ready.connect(self.tambah_item)
        self.worker.finished.connect(self.proses_selesai)
        self.worker.start()   # mulai thread — run() dieksekusi di background

    def tambah_item(self, item):
        self.list_widget.addItem(item)

    def proses_selesai(self):
        self.label.setText("Selesai!")
        self.btn_start.setEnabled(True)


if __name__ == "__main__":
    app = QApplication(sys.argv)
    w = GoodDemo()
    w.show()
    sys.exit(app.exec())

Race Condition — Risiko dari Threading

Threading memang membuat UI tetap responsif, tapi ada risiko yang perlu diwaspadai: race condition. Ini terjadi ketika dua thread mengakses dan memodifikasi data yang sama secara bersamaan, sehingga hasilnya tidak bisa diprediksi.

Analogi Sederhana

Bayangkan dua kasir di toko yang sama melihat stok barang = 1, lalu keduanya menjual barang itu ke pelanggan berbeda secara bersamaan. Hasilnya: stok jadi -1 karena keduanya tidak tahu satu sama lain sudah menjual.

Contoh berikut menunjukkan perbedaan antara eksekusi berurutan (aman) vs bersamaan dengan thread (berpotensi salah):

dengan_thread.py — Bersamaan, Bisa Salah RACE CONDITION 
import threading
import time

saldo = 100

def tarik_tunai(nama, jumlah):
    global saldo
    
    # STEP 1: Baca saldo saat ini
    saldo_dibaca = saldo
    print(f"[{nama}] Cek saldo: {saldo_dibaca}")
    
    # Simulasi jeda (misal: verifikasi PIN, koneksi jaringan)
    # Di sinilah thread lain bisa "menyela" dan mengubah saldo!
    time.sleep(0.1)
    
    # STEP 2: Cek dan kurangi — tapi saldo mungkin sudah berubah!
    if saldo_dibaca >= jumlah:
        saldo = saldo_dibaca - jumlah  # Pakai nilai lama yang sudah stale
        print(f"[{nama}] Berhasil tarik {jumlah}, saldo: {saldo}")
    else:
        print(f"[{nama}] Gagal, saldo tidak cukup")

# Dua thread berjalan BERSAMAAN mengakses saldo yang sama
t1 = threading.Thread(target=tarik_tunai, args=("ATM-1", 80))
t2 = threading.Thread(target=tarik_tunai, args=("ATM-2", 80))
t1.start()
t2.start()
t1.join()
t2.join()

# Output yang terjadi (SALAH!):
# [ATM-1] Cek saldo: 100  ← baca 100
# [ATM-2] Cek saldo: 100  ← juga baca 100, belum ada yang kurangi
# [ATM-1] Berhasil tarik 80, saldo: 20
# [ATM-2] Berhasil tarik 80, saldo: 20  ← harusnya ditolak!
print(f"Saldo akhir: {saldo}")  # 20, padahal uang sudah ditarik 160!
Kesimpulan: Threading Membawa Risiko Race Condition

Threading bukan solusi untuk race condition — justru threading yang menyebabkan race condition bisa terjadi. Tanpa thread, kode berjalan berurutan sehingga tidak ada masalah. Begitu kita pakai thread, kita harus waspada terhadap akses data bersama.

Di PySide6, solusinya adalah Signal: worker thread tidak boleh langsung mengubah data UI, melainkan mengirim Signal ke main thread, dan Qt yang mengurus eksekusinya secara aman.

Solusi Race Condition: threading.Lock()

Seperti di Java yang mengenal synchronized, Python menyediakan threading.Lock() untuk mengunci akses ke data bersama. Selama satu thread memegang lock, thread lain yang mencoba mengaksesnya akan menunggu sampai lock dilepas.

dengan_lock.py — Race Condition Teratasi AMAN 
import threading
import time

saldo = 100
lock = threading.Lock()   # buat satu lock yang dipakai bersama

def tarik_tunai(nama, jumlah):
    global saldo

    # with lock: → acquire lock saat masuk, release otomatis saat keluar
    # Thread lain yang sampai di baris ini akan MENUNGGU sampai lock bebas
    with lock:
        saldo_dibaca = saldo
        print(f"[{nama}] Cek saldo: {saldo_dibaca}")

        time.sleep(0.1)   # simulasi jeda — tapi thread lain tetap menunggu

        if saldo_dibaca >= jumlah:
            saldo = saldo_dibaca - jumlah
            print(f"[{nama}] Berhasil tarik {jumlah}, saldo: {saldo}")
        else:
            print(f"[{nama}] Gagal, saldo tidak cukup")

t1 = threading.Thread(target=tarik_tunai, args=("ATM-1", 80))
t2 = threading.Thread(target=tarik_tunai, args=("ATM-2", 80))
t1.start()
t2.start()
t1.join()
t2.join()

# Output yang benar (salah satu urutan yang mungkin):
# [ATM-1] Cek saldo: 100
# [ATM-1] Berhasil tarik 80, saldo: 20
# [ATM-2] Cek saldo: 20   ← ATM-2 baru bisa masuk setelah ATM-1 selesai
# [ATM-2] Gagal, saldo tidak cukup

GIL — Kenapa Race Condition Masih Bisa Terjadi di Python?

Python punya GIL (Global Interpreter Lock) — lock bawaan di level interpreter yang memastikan hanya satu thread yang mengeksekusi bytecode Python pada satu waktu. Sekilas ini terdengar seperti race condition tidak mungkin terjadi, tapi kenyataannya tetap bisa.

Mengapa GIL Tidak Cukup?

GIL bisa dilepas di antara instruksi bytecode — termasuk di tengah operasi yang terlihat sederhana. Baris saldo -= jumlah sebenarnya terdiri dari beberapa instruksi bytecode (baca, kurangi, tulis), dan GIL bisa berpindah ke thread lain di antaranya.

GIL juga dilepas saat operasi I/O dan time.sleep() — itulah mengapa contoh ATM di atas bisa mengalami race condition meski Python punya GIL.

Mekanisme Bahasa Cara Kerja Kapan Dipakai
synchronized Java Lock otomatis pada method atau blok kode Akses data bersama antar thread
threading.Lock() Python Lock manual dengan with lock: Akses data bersama antar thread
Signal PySide6 / Qt Kirim data via event queue, dieksekusi di main thread Update UI dari worker thread
GIL Python Lock global di level interpreter, bukan per-variabel Otomatis — tapi tidak cukup untuk data bersama
Lock vs Signal di PySide6 — Kapan Pakai yang Mana?
  • Signal — untuk update widget UI dari worker thread. Ini cara yang direkomendasikan Qt dan sudah thread-safe secara otomatis.
  • threading.Lock() — untuk melindungi data non-UI yang diakses dari beberapa thread sekaligus, misalnya list hasil, counter, atau dict cache yang dibaca/ditulis banyak worker.

Dalam aplikasi PySide6 sederhana, kita jarang perlu Lock secara eksplisit karena pola Worker + Signal sudah menghindari akses data bersama. Lock baru diperlukan saat ada beberapa worker thread yang berbagi data yang sama di luar UI.

Masalah Umum dalam Threading

Masalah Penjelasan Solusi
Race Condition Dua thread memodifikasi data yang sama bersamaan Gunakan Lock / Signal
Deadlock Dua thread saling menunggu satu sama lain selamanya Hindari nested lock, gunakan timeout
UI Update dari Thread Update widget langsung dari worker thread → crash/undefined behavior Selalu gunakan Signal untuk kirim data ke Main Thread
Dangling Thread Thread masih berjalan setelah window ditutup Hubungkan finished ke thread.quit
Mengapa Signal Aman untuk Update UI?

Qt's Signal-Slot mechanism secara otomatis menangani komunikasi antar thread. Ketika signal dikirim dari worker thread ke slot di main thread, Qt akan mengantri pesan tersebut dan mengeksekusinya di main thread melalui event loop — sehingga tidak ada race condition pada UI.

Ringkasan: Aturan Dasar Threading di PySide6

Aturan Penjelasan
Proses berat masuk ke run() Method run() dieksekusi di thread terpisah — main thread tetap bebas melayani UI
Update UI hanya via Signal Jangan langsung ubah widget dari dalam run(). Kirim data lewat emit(), biarkan slot di main thread yang mengubah widget
Simpan worker sebagai self.worker Variabel lokal akan dihapus setelah fungsi selesai. Simpan sebagai atribut agar thread tidak berhenti tiba-tiba

3. Progress Reporting & Cancellation

Untuk proses yang lama, user perlu melihat progress dan bisa membatalkan proses. Gunakan flag _is_cancelled pada worker.

cancel_pattern.py Pattern 
class CancellableWorker(QObject):
    finished = Signal()
    progress = Signal(int)   # Signal bertipe int — akan membawa angka progress
    result = Signal(str)
    
    def __init__(self):
        super().__init__()
        self._is_cancelled = False
    
    def cancel(self):
        """Dipanggil dari main thread untuk cancel"""
        self._is_cancelled = True
    
    def run(self):
        for i in range(100):
            # Cek apakah sudah di-cancel
            if self._is_cancelled:
                self.result.emit("Dibatalkan oleh user")
                break
            
            time.sleep(0.1)
            # emit(i + 1) → "kirimkan nilai i+1 ke semua slot yang terhubung"
            # i dimulai dari 0, jadi i+1 = 1,2,3,...,100 (lebih intuitif sebagai progress)
            # Slot yang menerima (misal: progress_bar.setValue) akan dipanggil dengan nilai ini
            self.progress.emit(i + 1)
        else:
            self.result.emit("Selesai!")
        
        self.finished.emit()

# Di MainWindow:
# self.cancel_btn.clicked.connect(self.worker.cancel)
Apa itu emit()?

emit() adalah cara worker thread mengirimkan data ke main thread melalui Signal. Analoginya seperti menekan tombol bel — siapa pun yang sudah "mendaftar" untuk mendengar bel itu (slot) akan dipanggil secara otomatis.

Baris Artinya
progress = Signal(int) Deklarasi: "Signal ini akan membawa satu nilai bertipe int"
self.progress.emit(i + 1) Kirim nilai i+1 ke semua slot yang terhubung ke signal ini
worker.progress.connect(progress_bar.setValue) Daftarkan setValue sebagai penerima — setiap emit akan memanggil setValue(nilai)

Jadi alurnya: worker memanggil self.progress.emit(45) → Qt meneruskan nilai 45 ke main thread → progress_bar.setValue(45) dieksekusi di main thread → progress bar bergerak ke 45%.

Nilai yang dikirim via emit() harus sesuai tipe dengan deklarasi Signal-nya. Signal(int) hanya bisa membawa int, Signal(str) hanya bisa membawa str, dan seterusnya.

7. API Call di Thread Terpisah

Di section 6 kita sudah bisa memanggil requests.get() dari terminal. Sekarang kita integrasikan ke PySide6 — tapi dengan satu aturan wajib: semua request harus berjalan di thread terpisah, bukan di main thread.

Aturan Wajib

Jangan pernah memanggil requests.get() langsung di dalam slot/handler PySide6. UI akan freeze sampai response datang. Selalu pindahkan ke QThread Worker.

Langkah 1 — Buat Worker untuk GET

Kita mulai dari yang paling sederhana: ambil daftar posts lalu tampilkan ke QTextEdit. Buat file baru bernama latihan_get.py dan ketik kode berikut:

latihan_get.py Langkah 1 — Ketik & Jalankan 
# Library yang dibutuhkan
import sys
import requests                          # untuk HTTP request ke API
from PySide6.QtWidgets import (
    QApplication, QWidget, QVBoxLayout,
    QPushButton, QLabel, QTextEdit
)
from PySide6.QtCore import QThread, Signal, QObject
                                              # QThread  : membuat thread baru
                                              # Signal   : mengirim data antar thread
                                              # QObject  : base class untuk worker

BASE_URL = "https://jsonplaceholder.typicode.com"


# ─────────────────────────────────────────────────────
# GetWorker: class yang menjalankan requests.get()
# di thread terpisah agar UI tidak freeze
# ─────────────────────────────────────────────────────
class GetWorker(QObject):

    # Signal dikirim ke MainWindow setelah run() selesai
    finished = Signal()              # selalu dipanggil di akhir (sukses/gagal)
    result   = Signal(object)   # membawa data JSON (list/dict)
    error    = Signal(str)      # membawa pesan error jika gagal

    def __init__(self, url):
        super().__init__()
        self.url = url   # simpan URL yang akan di-request

    def run(self):
        # method ini dijalankan oleh QThread, bukan main thread
        try:
            response = requests.get(self.url, timeout=10)
            response.raise_for_status()          # error jika status 4xx/5xx
            self.result.emit(response.json())  # kirim data ke MainWindow
        except Exception as e:
            self.error.emit(str(e))   # kirim pesan error ke MainWindow
        finally:
            self.finished.emit()           # selalu emit agar thread bisa berhenti


# ─────────────────────────────────────────────────────
# MainWindow: tampilan UI dan pengatur thread
# ─────────────────────────────────────────────────────
class MainWindow(QWidget):
    def __init__(self):
        super().__init__()
        self.setWindowTitle("Latihan GET")
        layout = QVBoxLayout(self)

        self.status = QLabel("Klik tombol untuk fetch posts")
        layout.addWidget(self.status)

        self.btn = QPushButton("Fetch Posts")
        self.btn.clicked.connect(self.fetch_posts)
        layout.addWidget(self.btn)

        self.output = QTextEdit()
        self.output.setReadOnly(True)
        layout.addWidget(self.output)

    def fetch_posts(self):
        # 1. Update UI ke state loading
        self.status.setText("Loading...")
        self.btn.setEnabled(False)  # disable agar tidak diklik dua kali

        # 2. Buat worker dan thread
        self.thread = QThread()
        self.worker = GetWorker(f"{BASE_URL}/posts")
        self.worker.moveToThread(self.thread)  # pindahkan worker ke thread

        # 3. Hubungkan signals ke slots
        self.thread.started.connect(self.worker.run)      # thread mulai → jalankan run()
        self.worker.result.connect(self.on_result)      # data tiba → on_result()
        self.worker.error.connect(self.on_error)        # error → on_error()
        self.worker.finished.connect(self.thread.quit)   # selesai → hentikan thread
        self.worker.finished.connect(lambda: self.btn.setEnabled(True))  # aktifkan tombol kembali

        # 4. Mulai thread
        self.thread.start()

    def on_result(self, posts):
        # dipanggil oleh signal result — data sudah ada di sini
        self.status.setText(f"{len(posts)} posts dimuat")
        text = ""
        for post in posts[:10]:  # tampilkan 10 pertama
            text += f"[{post['id']}] {post['title']}\n"
        self.output.setText(text)

    def on_error(self, msg):
        # dipanggil oleh signal error — tampilkan pesan ke user
        self.status.setText(f"Error: {msg}")

if __name__ == "__main__":
    app = QApplication(sys.argv)
    w = MainWindow()
    w.show()
    sys.exit(app.exec())

8. State Handling: Loading, Success, Error, Empty

Setiap API call punya 4 kemungkinan kondisi. Aplikasi yang baik harus menangani semuanya. Kita tambahkan state handling langsung ke latihan_get.py dari langkah sebelumnya.

10. Aplikasi API Client Lengkap dengan Separation of Concerns

Pada bagian ini kita membangun aplikasi API client yang sama, tetapi dengan struktur multi-file. Tujuannya agar mahasiswa tidak terbiasa menaruh service, worker, dialog, dan window utama dalam satu file besar.

Apa itu Separation of Concerns (SoC)?

SoC adalah prinsip memisahkan program berdasarkan tanggung jawab. File untuk HTTP request sebaiknya tidak berisi widget. File untuk dialog sebaiknya tidak mengurus network. Main window bertugas mengorkestrasi alur, bukan memuat semua detail implementasi.

Struktur File Proyek

Untuk materi kelas, struktur berikut sudah cukup rapi dan realistis:

# Tanggung jawab: semua logika HTTP request ke JSONPlaceholder API
# Tidak ada Qt di sini — bisa diuji tanpa membuka UI

import requests

class ApiService:
    """
    Service class untuk semua interaksi HTTP dengan JSONPlaceholder.
    Setiap method melempar exception jika request gagal —
    exception ini akan ditangkap oleh ApiWorker di layer atas.
    """
    
    BASE_URL = "https://jsonplaceholder.typicode.com"
    TIMEOUT = 10  # batas waktu tunggu response (detik)
    
    # ===== POSTS =====
    
    def get_posts(self):
        """GET /posts - Ambil semua posts (100 data)"""
        response = requests.get(
            f"{self.BASE_URL}/posts",
            timeout=self.TIMEOUT
        )
        response.raise_for_status()  # lempar exception jika status 4xx/5xx
        return response.json()         # kembalikan list of dict
    
    def get_comments(self, post_id):
        """GET /posts/{id}/comments - Ambil komentar dari 1 post"""
        response = requests.get(
            f"{self.BASE_URL}/posts/{post_id}/comments",
            timeout=self.TIMEOUT
        )
        response.raise_for_status()
        return response.json()
    
    def create_post(self, title, body, user_id=1):
        """POST /posts - Buat post baru
        
        Mengirim data JSON ke server. Server akan mengembalikan
        data yang sama ditambah 'id' baru (selalu 101 di JSONPlaceholder).
        """
        payload = {
            'title': title,
            'body': body,
            'userId': user_id
        }
        response = requests.post(
            f"{self.BASE_URL}/posts",
            json=payload,       # otomatis set Content-Type: application/json
            timeout=self.TIMEOUT
        )
        response.raise_for_status()
        return response.json()  # {'id': 101, 'title': ..., 'body': ..., 'userId': 1}
    
    def update_post(self, post_id, title, body, user_id=1):
        """PUT /posts/{id} - Update seluruh data post"""
        payload = {
            'id': post_id,
            'title': title,
            'body': body,
            'userId': user_id
        }
        response = requests.put(
            f"{self.BASE_URL}/posts/{post_id}",
            json=payload,
            timeout=self.TIMEOUT
        )
        response.raise_for_status()
        return response.json()
    
    def delete_post(self, post_id):
        """DELETE /posts/{id} - Hapus post
        
        JSONPlaceholder mengembalikan {} saat delete berhasil.
        Kita kembalikan True sebagai tanda sukses agar lebih eksplisit.
        """
        response = requests.delete(
            f"{self.BASE_URL}/posts/{post_id}",
            timeout=self.TIMEOUT
        )
        response.raise_for_status()
        return True
    
    # ===== COMMENTS =====
    
    def get_comments_by_post(self, post_id):
        """GET /posts/{id}/comments - Ambil comments dari post"""
        response = requests.get(
            f"{self.BASE_URL}/posts/{post_id}/comments",
            timeout=self.TIMEOUT
        )
        response.raise_for_status()
        return response.json()

Mengapa raise_for_status() di Setiap Method?

raise_for_status() otomatis melempar HTTPError jika server mengembalikan status 4xx atau 5xx. Dengan ini, kita tidak perlu menulis if response.status_code != 200 di setiap method — cukup satu try/except di ApiWorker untuk menangkap semua kemungkinan error.

Poin Penting SoC di File Ini

ApiService tidak tahu apakah ia dipanggil dari thread atau dari main thread. Ia tidak tahu ada UI. Tugasnya hanya satu: kirim request, kembalikan data. Keputusan "di thread mana dijalankan" adalah urusan ApiWorker di file berikutnya.

File 2: api_worker.py — Layer Threading

Worker bertugas menjalankan method dari ApiService di thread terpisah menggunakan QThread. File ini adalah jembatan antara layer data dan layer UI — ia tidak tahu cara menampilkan data, dan tidak tahu cara mengambil data dari internet. Tugasnya hanya: jalankan di thread lain, kirim hasilnya via Signal.

Mengapa ApiWorker Perlu File Sendiri?

Jika worker digabung di main.py, setiap kali kita ingin menambah aksi baru (misal: fetch users), kita harus mengubah file UI. Dengan memisahkannya, main.py cukup memanggil ApiWorker("get_users", ...) tanpa tahu detail implementasinya.

api_worker.py Layer Threading 
# api_worker.py
# Tanggung jawab: menjalankan ApiService di thread terpisah
# Tidak ada logika HTTP di sini, tidak ada widget UI

from PySide6.QtCore import QObject, Signal
from api_service import ApiService


class ApiWorker(QObject):
    """
    Worker yang berjalan di QThread terpisah.
    Menerima nama aksi (action) dan parameter yang dibutuhkan,
    lalu memanggil method yang sesuai di ApiService.
    """
    
    # Signal untuk komunikasi balik ke Main Thread
    finished = Signal()           # selalu dipanggil di akhir (sukses maupun gagal)
    success = Signal(object)      # kirim data hasil (list/dict/bool)
    error = Signal(str)          # kirim pesan error
    
    def __init__(self, action, post_id=None, title=None, body=None):
        super().__init__()
        self.action = action      # nama aksi: 'get_posts', 'create_post', dst
        self.post_id = post_id    # dipakai untuk get_comments, update, delete
        self.title = title        # dipakai untuk create_post dan update_post
        self.body = body          # dipakai untuk create_post dan update_post
        self.service = ApiService()  # instance service dibuat di sini
    
    def run(self):
        """Dijalankan oleh QThread. Panggil service sesuai action."""
        try:
            if self.action == "get_posts":
                result = self.service.get_posts()
            
            elif self.action == "get_comments":
                result = self.service.get_comments_by_post(self.post_id)
            
            elif self.action == "create_post":
                result = self.service.create_post(self.title, self.body)
            
            elif self.action == "update_post":
                result = self.service.update_post(self.post_id, self.title, self.body)
            
            elif self.action == "delete_post":
                result = self.service.delete_post(self.post_id)
            
            else:
                raise ValueError(f"Action tidak dikenali: {self.action}")
            
            self.success.emit(result)  # kirim hasil ke main thread
        
        except Exception as e:
            self.error.emit(str(e))   # kirim pesan error ke main thread
        
        finally:
            self.finished.emit()       # selalu emit finished agar thread bisa berhenti
Mengapa finally: self.finished.emit()?

Blok finally selalu dieksekusi, baik request berhasil maupun gagal. Ini memastikan finished signal selalu dikirim sehingga thread selalu berhenti dengan bersih — tidak ada dangling thread yang terus berjalan di background.

File 3: dialogs.py — Layer Komponen UI

Dialog dipisah dari main.py agar window utama tidak dipenuhi kode form input. Jika nanti ada banyak dialog (tambah user, edit komentar, dll), semuanya cukup ditaruh di file ini tanpa menyentuh main.py.

Kapan Dialog Muncul?

PostDialog dipakai dua kali: saat Tambah (tanpa data awal) dan saat Edit (dengan data post yang sudah ada). Parameter post=None mengontrol perilaku ini — jika diisi, form akan terisi otomatis dengan data lama.

        self.body_input.setMaximumHeight(150)

        form.addRow("Body:", self.body_input)

Validasi Input

Perhatikan .strip() di get_data() — ini membuang spasi di awal/akhir agar tidak mengirim data kosong ke API. Validasi lebih lanjut (cek apakah title kosong) sebaiknya dilakukan di main.py sebelum memanggil worker.

File 4: main.py — Window Utama

File ini adalah orkestrator — ia menghubungkan semua komponen. Tidak ada logika HTTP, tidak ada form input langsung. main.py hanya tahu cara: meminta data (lewat worker), menampilkan data (ke tabel/label), dan merespons aksi user (klik tombol).

Pola run_worker()

Daripada menulis boilerplate QThread + moveToThread + connect signals berulang kali, kita buat satu method run_worker() yang menangani semuanya. Setiap kali butuh API call, cukup panggil self.run_worker("nama_aksi", callback).

main.py Main Window 
# main.py
# Tanggung jawab: UI utama, event handler, orkestrasi semua komponen
# Import dari file lain — tidak ada requests, tidak ada form detail

import sys
from PySide6.QtCore import QThread
from PySide6.QtWidgets import (
    QApplication, QMainWindow, QWidget, QVBoxLayout, QHBoxLayout,
    QPushButton, QLabel, QTableWidget, QTableWidgetItem,
    QTextEdit, QSplitter, QHeaderView, QMessageBox, QDialog
)
from PySide6.QtCore import Qt
from dialogs import PostDialog      # ← dari dialogs.py
from api_worker import ApiWorker    # ← dari api_worker.py


class PostViewerApp(QMainWindow):
    def __init__(self):
        super().__init__()
        self.setWindowTitle("API Client — SoC Demo")
        self.setGeometry(100, 100, 900, 600)
        self.posts_data = []
        self._thread = None
        self._worker = None
        self.setup_ui()
        self.fetch_posts()  # langsung fetch saat aplikasi dibuka
    
    # ===== SETUP UI =====
    
    def setup_ui(self):
        central = QWidget()
        self.setCentralWidget(central)
        layout = QVBoxLayout(central)
        
        # Status bar atas
        self.status_label = QLabel("Ready")
        self.status_label.setStyleSheet("font-weight: bold; padding: 4px;")
        layout.addWidget(self.status_label)
        
        # Tombol aksi
        btn_layout = QHBoxLayout()
        self.btn_refresh = QPushButton("Refresh")
        self.btn_tambah  = QPushButton("Tambah")
        self.btn_edit    = QPushButton("Edit")
        self.btn_hapus   = QPushButton("Hapus")
        btn_layout.addWidget(self.btn_refresh)
        btn_layout.addWidget(self.btn_tambah)
        btn_layout.addWidget(self.btn_edit)
        btn_layout.addWidget(self.btn_hapus)
        layout.addLayout(btn_layout)
        
        # Splitter: tabel kiri | detail kanan
        splitter = QSplitter(Qt.Horizontal)
        
        self.table = QTableWidget()
        self.table.setColumnCount(2)
        self.table.setHorizontalHeaderLabels(['ID', 'Title'])
        self.table.setSelectionBehavior(QTableWidget.SelectRows)
        self.table.setEditTriggers(QTableWidget.NoEditTriggers)
        self.table.horizontalHeader().setSectionResizeMode(1, QHeaderView.Stretch)
        self.table.setColumnWidth(0, 40)
        splitter.addWidget(self.table)
        
        self.detail = QTextEdit()
        self.detail.setReadOnly(True)
        splitter.addWidget(self.detail)
        splitter.setSizes([400, 500])
        layout.addWidget(splitter)
        
        # Hubungkan signals ke slots
        self.btn_refresh.clicked.connect(self.fetch_posts)
        self.btn_tambah.clicked.connect(self.add_post)
        self.btn_edit.clicked.connect(self.edit_post)
        self.btn_hapus.clicked.connect(self.delete_post)
        self.table.currentCellChanged.connect(self.on_row_selected)
    
    # ===== HELPER: JALANKAN WORKER =====
    
    def run_worker(self, action, on_success, post_id=None, title=None, body=None):
        """Helper: buat QThread + ApiWorker, hubungkan signals, lalu start."""
        self._thread = QThread()
        self._worker = ApiWorker(action, post_id=post_id, title=title, body=body)
        self._worker.moveToThread(self._thread)
        
        self._thread.started.connect(self._worker.run)
        self._worker.success.connect(on_success)
        self._worker.error.connect(self.on_error)
        self._worker.finished.connect(self._thread.quit)
        self._worker.finished.connect(lambda: self.set_loading(False))
        
        self.set_loading(True)
        self._thread.start()
    
    def set_loading(self, is_loading):
        """Aktifkan/nonaktifkan tombol saat request berjalan."""
        for btn in [self.btn_refresh, self.btn_tambah, self.btn_edit, self.btn_hapus]:
            btn.setEnabled(not is_loading)
        if is_loading:
            self.status_label.setText("Loading...")
            self.status_label.setStyleSheet("font-weight:bold; color:#3498db; padding:4px;")
    
    # ===== FETCH POSTS =====
    
    def fetch_posts(self):
        self.run_worker("get_posts", self.on_posts_loaded)
    
    def on_posts_loaded(self, posts):
        self.posts_data = posts[:20]
        self.table.setRowCount(0)
        
        for post in self.posts_data:
            row = self.table.rowCount()
            self.table.insertRow(row)
            self.table.setItem(row, 0, QTableWidgetItem(str(post['id'])))
            self.table.setItem(row, 1, QTableWidgetItem(post['title']))
        
        self.status_label.setText(f"{len(posts)} posts dimuat (menampilkan 20)")
        self.status_label.setStyleSheet("font-weight:bold; color:#27ae60; padding:4px;")
    
    # ===== ROW SELECTED → TAMPILKAN DETAIL =====
    
    def on_row_selected(self, row, col, prev_row, prev_col):
        if row < 0 or row >= len(self.posts_data):
            return
        post = self.posts_data[row]
        self.detail.setPlainText(
            f"ID    : {post['id']}\n"
            f"Title : {post['title']}\n\n"
            f"Body  :\n{post['body']}"
        )
    
    # ===== ADD POST =====
    
    def add_post(self):
        dialog = PostDialog(self)  # buka dialog kosong (mode Tambah)
        if dialog.exec() == QDialog.Accepted:
            data = dialog.get_data()
            if not data['title'] or not data['body']:
                QMessageBox.warning(self, "Validasi", "Title dan Body wajib diisi!")
                return
            self.run_worker(
                "create_post", self.on_post_created,
                title=data['title'], body=data['body']
            )
    
    def on_post_created(self, result):
        QMessageBox.information(self, "Sukses", f"Post dibuat! ID: {result['id']}")
        self.fetch_posts()
    
    # ===== EDIT POST =====
    
    def edit_post(self):
        row = self.table.currentRow()
        if row < 0:
            QMessageBox.warning(self, "Peringatan", "Pilih post terlebih dahulu!")
            return
        post = self.posts_data[row]
        dialog = PostDialog(self, post)  # buka dialog dengan data lama (mode Edit)
        if dialog.exec() == QDialog.Accepted:
            data = dialog.get_data()
            self.run_worker(
                "update_post", self.on_post_updated,
                post_id=post['id'], title=data['title'], body=data['body']
            )
    
    def on_post_updated(self, result):
        QMessageBox.information(self, "Sukses", "Post berhasil diupdate!")
        self.fetch_posts()
    
    # ===== DELETE POST =====
    
    def delete_post(self):
        row = self.table.currentRow()
        if row < 0:
            QMessageBox.warning(self, "Peringatan", "Pilih post terlebih dahulu!")
            return
        post_id = self.posts_data[row]['id']
        reply = QMessageBox.question(
            self, "Konfirmasi",
            f"Yakin ingin menghapus post ID {post_id}?",
            QMessageBox.Yes | QMessageBox.No
        )
        if reply == QMessageBox.Yes:
            self.run_worker("delete_post", self.on_post_deleted, post_id=post_id)
    
    def on_post_deleted(self, result):
        QMessageBox.information(self, "Sukses", "Post berhasil dihapus!")
        self.fetch_posts()
    
    # ===== ERROR HANDLER =====
    
    def on_error(self, message):
        self.status_label.setText(f"Error: {message}")
        self.status_label.setStyleSheet("font-weight:bold; color:#e74c3c; padding:4px;")
        QMessageBox.critical(self, "API Error", message)

  • Fetch Posts: data diambil oleh ApiService, dijalankan oleh ApiWorker, ditampilkan oleh main.py
  • View Detail: klik baris tabel → tampilkan detail di panel kanan
  • Add Post: input di PostDialog, request dikirim lewat worker
  • Edit Post: form terisi data lama, update dikirim lewat worker
  • Delete Post: konfirmasi dulu, baru hapus lewat worker
  • State Loading: semua tombol disabled saat request berjalan
Kenapa Struktur Ini Lebih Baik?
  • Mudah dibaca: mahasiswa tahu file mana untuk network, file mana untuk UI
  • Mudah dikembangkan: ingin tambah endpoint /users? Cukup tambah method di api_service.py
  • Mudah diuji: ApiService bisa diuji tanpa membuka GUI sama sekali
  • Mudah dirawat: error lebih mudah dilacak karena tanggung jawab tidak bercampur

Tugas Praktikum

Tugas: Users Manager

Buatlah aplikasi desktop untuk mengelola data users dari JSONPlaceholder API (https://jsonplaceholder.typicode.com/users). Aplikasi harus mampu melakukan operasi CRUD lengkap dengan UI yang responsif.

Catatan: JSONPlaceholder adalah Fake API

JSONPlaceholder menerima semua request dan mengembalikan response yang valid, tetapi data tidak benar-benar tersimpan di server. POST selalu mengembalikan id: 11, PUT mengembalikan data yang dikirim, DELETE mengembalikan {} — semuanya simulasi. Ini normal dan memang tujuannya untuk latihan. Jangan bingung jika setelah DELETE, data masih muncul saat di-GET ulang.

Fitur yang Harus Diimplementasikan:

  1. Tampilkan Daftar Users — ambil semua users via GET, tampilkan dalam tabel dengan kolom: No, Nama, Username, Email, Kota
  2. Detail User — klik baris tabel untuk menampilkan detail lengkap user (nama, email, telepon, website, perusahaan) di panel samping
  3. Tambah User — form input untuk POST user baru (nama, username, email), tampilkan ID yang dikembalikan server
  4. Edit User — pilih user dari tabel, ubah data via PUT, tampilkan konfirmasi sukses
  5. Hapus User — pilih user dari tabel, hapus via DELETE dengan konfirmasi dialog terlebih dahulu
  6. Threading — semua API call harus berjalan di thread terpisah, UI tidak boleh freeze
  7. State Handling — tampilkan status loading saat request berjalan, pesan error jika gagal
Ketentuan
  • Endpoint yang digunakan: /users dan /users/{id}
  • Implementasikan semua 4 HTTP method: GET, POST, PUT, DELETE
  • Semua network request HARUS di thread terpisah — tidak boleh ada UI freeze
  • Implementasikan error handling: timeout dan connection error
  • Tombol Edit dan Hapus hanya aktif jika ada baris yang dipilih di tabel
