Praktikum: Penyimpanan Data di Android

Tujuan Pembelajaran

  • Memahami berbagai metode penyimpanan data di Android
  • Mampu mengimplementasikan SharedPreferences untuk data key-value sederhana
  • Mampu menyimpan dan membaca file di App-Specific Storage
  • Memahami konsep SQLite dan operasi CRUD dasar
  • Mampu mengimplementasikan Room Database sebagai abstraksi SQLite
  • Dapat memilih metode penyimpanan yang tepat sesuai kebutuhan

Prasyarat

Pastikan Anda sudah memahami materi dari praktikum sebelumnya:

  • Activity, Fragment, dan Navigation Component
  • RecyclerView dan Adapter
  • Layout XML dan Basic Widgets
  • Kotlin dasar (data class, lambda, coroutines dasar)

Mengapa Penyimpanan Data Penting?

Secara default, data dalam aplikasi Android bersifat sementara (volatile). Ketika aplikasi ditutup atau device di-restart, semua data di memori akan hilang. Penyimpanan data (data persistence) memungkinkan aplikasi menyimpan data secara permanen sehingga tetap tersedia meskipun aplikasi ditutup.

Lokasi Penyimpanan di Device

  • SharedPreferences: /data/data/com.example.app/shared_prefs/
  • Internal Storage: /data/data/com.example.app/files/
  • SQLite/Room: /data/data/com.example.app/databases/
  • Cache: /data/data/com.example.app/cache/

Semua lokasi di atas bersifat private — hanya aplikasi Anda yang bisa mengaksesnya. Data akan dihapus saat aplikasi di-uninstall.

SharedPreferences

Pengenalan SharedPreferences

SharedPreferences adalah mekanisme penyimpanan data sederhana yang menyimpan data dalam format key-value pairs. Data disimpan dalam file XML di internal storage aplikasi.

Karakteristik SharedPreferences

  • Menyimpan data primitif: String, Int, Long, Float, Boolean, Set<String>
  • Data tersimpan secara persisten (tetap ada setelah app ditutup)
  • Data private — hanya bisa diakses oleh aplikasi Anda
  • Operasi bersifat synchronous (blocking)
  • Cocok untuk data kecil dan sederhana
  • Tidak cocok untuk data besar atau data yang sering berubah

Format Penyimpanan

File XML yang dibuat oleh SharedPreferences
<!-- /data/data/com.example.app/shared_prefs/MyPrefs.xml -->
<?xml version='1.0' encoding='utf-8' standalone='yes' ?>
<map>
    <string name="username">john_doe</string>
    <boolean name="is_logged_in" value="true" />
    <int name="login_count" value="5" />
    <string name="theme">dark</string>
</map>

Cara Mendapatkan Instance SharedPreferences

Mendapatkan SharedPreferences
// Cara 1: SharedPreferences dengan nama spesifik
// Bisa diakses dari Activity manapun dengan nama yang sama
val sharedPref = getSharedPreferences("MyAppPrefs", Context.MODE_PRIVATE)

// Cara 2: SharedPreferences default untuk Activity ini saja
// Nama file otomatis berdasarkan nama Activity
val sharedPref = getPreferences(Context.MODE_PRIVATE)

// Cara 3: Di dalam Fragment
val sharedPref = requireContext().getSharedPreferences("MyAppPrefs", Context.MODE_PRIVATE)

Mode SharedPreferences

Parameter kedua getSharedPreferences() menentukan mode akses file. Berikut semua mode yang pernah ada di Android:

Mode Status Keterangan
MODE_PRIVATE Gunakan ini File hanya bisa diakses oleh aplikasi sendiri. Satu-satunya pilihan yang aman dan valid di Android modern.
MODE_WORLD_READABLE Deprecated API 17 Mengizinkan aplikasi lain membaca file SharedPreferences Anda. Dihapus karena membuka celah kebocoran data sensitif.
MODE_WORLD_WRITEABLE Deprecated API 17 Mengizinkan aplikasi lain menulis ke file SharedPreferences Anda. Sangat berbahaya karena data bisa dimanipulasi app lain.
MODE_MULTI_PROCESS Deprecated API 23 Dirancang untuk skenario multi-process, namun tidak reliable dan menyebabkan race condition. Gunakan ContentProvider sebagai gantinya.

Untuk berbagi data antar aplikasi secara aman di Android modern, gunakan ContentProvider. Untuk menyimpan data sensitif (token, password), gunakan EncryptedSharedPreferences dari library Jetpack Security.

Menulis Data ke SharedPreferences

Untuk menulis data, gunakan objek SharedPreferences.Editor. Setiap operasi tulis harus diakhiri dengan apply() atau commit().

Menulis berbagai tipe data
val sharedPref = getSharedPreferences("MyAppPrefs", Context.MODE_PRIVATE)

// Mendapatkan Editor
val editor = sharedPref.edit()

// Menulis berbagai tipe data
editor.putString("username", "john_doe")
editor.putInt("user_age", 25)
editor.putBoolean("is_logged_in", true)
editor.putFloat("rating", 4.5f)
editor.putLong("last_login", System.currentTimeMillis())
editor.putStringSet("hobbies", setOf("Reading", "Coding", "Gaming"))

// Simpan perubahan
editor.apply()  // Asynchronous (recommended)
// atau
editor.commit()  // Synchronous (mengembalikan boolean)

Cara Ringkas dengan apply {}

Cara ringkas menulis data
val sharedPref = getSharedPreferences("MyAppPrefs", Context.MODE_PRIVATE)

// Menggunakan apply block (lebih ringkas)
sharedPref.edit().apply {
    putString("username", "john_doe")
    putInt("user_age", 25)
    putBoolean("is_logged_in", true)
    apply()  // Jangan lupa apply() di akhir
}

Perbedaan apply() vs commit()

Keduanya berfungsi menyimpan perubahan ke file SharedPreferences, tapi cara kerjanya berbeda.

apply() menyimpan perubahan secara asynchronous — artinya perintah simpan langsung diterima dan dijalankan di background thread, sementara kode berikutnya di main thread terus berjalan tanpa menunggu. Hasilnya tidak dikembalikan, jadi Anda tidak tahu apakah berhasil atau gagal. Karena tidak memblokir main thread, UI tetap responsif dan tidak ada risiko lag.

commit() menyimpan secara synchronous — kode akan berhenti dan menunggu sampai proses tulis ke disk selesai, baru kemudian melanjutkan ke baris berikutnya. Ia mengembalikan nilai Boolean: true jika berhasil, false jika gagal. Karena berjalan di main thread dan harus menunggu operasi disk, ini bisa menyebabkan UI terasa lambat jika dipanggil terlalu sering.

Aspek apply() commit()
Cara kerja Simpan di background, kode lanjut jalan Tunggu sampai selesai, baru lanjut
Return value void — tidak ada Boolean — sukses/gagal
Efek ke UI Tidak memblokir, UI tetap lancar Memblokir main thread sementara
Kapan pakai Hampir semua kasus Saat perlu konfirmasi berhasil/gagal

Mana yang lebih baik?

Gunakan apply() untuk hampir semua kasus. SharedPreferences dirancang untuk data kecil yang jarang gagal disimpan, sehingga nilai return dari commit() jarang benar-benar dibutuhkan.

Gunakan commit() hanya jika ada alasan spesifik untuk memastikan data sudah tersimpan sebelum melanjutkan — misalnya sebelum aplikasi langsung ditutup paksa (System.exit()) atau sebelum berpindah ke proses lain yang langsung membaca data tersebut.

Membaca Data dari SharedPreferences

Untuk membaca data, gunakan method get...(key, defaultValue). Parameter defaultValue akan dikembalikan jika key tidak ditemukan.

Membaca berbagai tipe data
val sharedPref = getSharedPreferences("MyAppPrefs", Context.MODE_PRIVATE)

// Membaca data dengan default value
val username = sharedPref.getString("username", "Guest")        // Default: "Guest"
val age = sharedPref.getInt("user_age", 0)                      // Default: 0
val isLoggedIn = sharedPref.getBoolean("is_logged_in", false)   // Default: false
val rating = sharedPref.getFloat("rating", 0.0f)                // Default: 0.0
val lastLogin = sharedPref.getLong("last_login", 0L)            // Default: 0
val hobbies = sharedPref.getStringSet("hobbies", emptySet())   // Default: emptySet

// Mengecek apakah key ada
val hasUsername = sharedPref.contains("username")  // true/false

// Mengambil semua data
val allData = sharedPref.all  // Map<String, *>

Penting: Default Value

Selalu berikan default value yang masuk akal. Jika key belum pernah disimpan, default value akan digunakan. Ini mencegah crash akibat data null atau tidak terduga.

Menghapus Data dari SharedPreferences

Menghapus data
val sharedPref = getSharedPreferences("MyAppPrefs", Context.MODE_PRIVATE)

// Menghapus satu key
sharedPref.edit().apply {
    remove("username")
    apply()
}

// Menghapus SEMUA data
sharedPref.edit().apply {
    clear()
    apply()
}

Contoh Lengkap: Fitur Remember Me (Login)

Fitur "Remember Me" memungkinkan user tidak perlu login ulang setiap kali membuka aplikasi. SharedPreferences digunakan di sini karena data yang disimpan bersifat sederhana (key-value), ukurannya kecil, dan perlu tetap ada meskipun aplikasi ditutup.

Mengapa SharedPreferences untuk kasus ini?

Saat user login dan mencentang "Remember Me", kita perlu menyimpan tiga hal: status login, nama user, dan flag remember me. Ketiganya adalah data primitif sederhana yang tidak memerlukan struktur tabel atau query kompleks. SharedPreferences adalah pilihan paling ringan dan tepat — tidak perlu setup database, tidak perlu dependency tambahan, dan bisa dibaca dalam satu baris kode kapanpun dibutuhkan.

Tanpa SharedPreferences, setiap kali user membuka app mereka harus login ulang karena data di memori hilang saat app ditutup.

utils/PrefManager.kt
package com.example.myapp.utils

import android.content.Context
import android.content.SharedPreferences

class PrefManager(context: Context) {

    private val sharedPref: SharedPreferences =
        context.getSharedPreferences(PREF_NAME, Context.MODE_PRIVATE)

    companion object {
        private const val PREF_NAME = "LoginPrefs"
        private const val KEY_IS_LOGGED_IN = "is_logged_in"
        private const val KEY_USERNAME = "username"
        private const val KEY_REMEMBER_ME = "remember_me"
    }

    // ===== SIMPAN STATUS LOGIN =====
    fun saveLoginSession(username: String, rememberMe: Boolean) {
        sharedPref.edit().apply {
            putBoolean(KEY_IS_LOGGED_IN, true)
            putString(KEY_USERNAME, username)
            putBoolean(KEY_REMEMBER_ME, rememberMe)
            apply()
        }
    }

    // ===== CEK STATUS LOGIN =====
    fun isLoggedIn(): Boolean {
        return sharedPref.getBoolean(KEY_IS_LOGGED_IN, false)
    }

    fun isRememberMe(): Boolean {
        return sharedPref.getBoolean(KEY_REMEMBER_ME, false)
    }

    // ===== AMBIL DATA USER =====
    fun getUsername(): String {
        return sharedPref.getString(KEY_USERNAME, "") ?: ""
    }

    // ===== LOGOUT (HAPUS SESSION) =====
    fun logout() {
        sharedPref.edit().apply {
            clear()
            apply()
        }
    }
}

Implementasi LoginActivity

Di sinilah SharedPreferences bekerja. Saat Activity dibuka, kita langsung cek apakah user sudah pernah login dengan Remember Me aktif. Jika ya, skip halaman login dan langsung ke MainActivity. Ini adalah keuntungan utama SharedPreferences — data tersedia instan tanpa query database.

LoginActivity.kt
package com.example.myapp

import android.content.Intent
import android.os.Bundle
import android.widget.Button
import android.widget.CheckBox
import android.widget.EditText
import android.widget.Toast
import androidx.appcompat.app.AppCompatActivity
import com.example.myapp.utils.PrefManager

class LoginActivity : AppCompatActivity() {

    private lateinit var prefManager: PrefManager

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        prefManager = PrefManager(this)

        // Cek apakah user sudah login dan remember me aktif
        if (prefManager.isLoggedIn() && prefManager.isRememberMe()) {
            // Langsung ke MainActivity, skip login
            navigateToMain()
            return
        }

        setContentView(R.layout.activity_login)

        val etUsername = findViewById<EditText>(R.id.etUsername)
        val etPassword = findViewById<EditText>(R.id.etPassword)
        val cbRememberMe = findViewById<CheckBox>(R.id.cbRememberMe)
        val btnLogin = findViewById<Button>(R.id.btnLogin)

        // Jika sebelumnya remember me aktif, isi username
        if (prefManager.isRememberMe()) {
            etUsername.setText(prefManager.getUsername())
            cbRememberMe.isChecked = true
        }

        btnLogin.setOnClickListener {
            val username = etUsername.text.toString().trim()
            val password = etPassword.text.toString().trim()

            if (username.isEmpty() || password.isEmpty()) {
                Toast.makeText(this, "Username dan password harus diisi", Toast.LENGTH_SHORT).show()
                return@setOnClickListener
            }

            // Simulasi validasi login (di real app, cek ke server/database)
            if (username == "admin" && password == "123456") {
                // Simpan session
                prefManager.saveLoginSession(
                    username = username,
                    rememberMe = cbRememberMe.isChecked
                )
                Toast.makeText(this, "Login berhasil!", Toast.LENGTH_SHORT).show()
                navigateToMain()
            } else {
                Toast.makeText(this, "Username atau password salah", Toast.LENGTH_SHORT).show()
            }
        }
    }

    private fun navigateToMain() {
        startActivity(Intent(this, MainActivity::class.java))
        finish()  // Tutup LoginActivity agar tidak bisa back
    }
}

Contoh: App Settings

SharedPreferences sangat cocok untuk menyimpan pengaturan aplikasi seperti tema, ukuran font, dan notifikasi.

utils/SettingsManager.kt
package com.example.myapp.utils

import android.content.Context
import android.content.SharedPreferences

class SettingsManager(context: Context) {

    private val prefs: SharedPreferences =
        context.getSharedPreferences("AppSettings", Context.MODE_PRIVATE)

    companion object {
        private const val KEY_DARK_MODE = "dark_mode"
        private const val KEY_FONT_SIZE = "font_size"
        private const val KEY_NOTIFICATION = "notification_enabled"
        private const val KEY_LANGUAGE = "language"
    }

    // Dark Mode
    var isDarkMode: Boolean
        get() = prefs.getBoolean(KEY_DARK_MODE, false)
        set(value) = prefs.edit().putBoolean(KEY_DARK_MODE, value).apply()

    // Font Size
    var fontSize: Int
        get() = prefs.getInt(KEY_FONT_SIZE, 14)
        set(value) = prefs.edit().putInt(KEY_FONT_SIZE, value).apply()

    // Notification
    var isNotificationEnabled: Boolean
        get() = prefs.getBoolean(KEY_NOTIFICATION, true)
        set(value) = prefs.edit().putBoolean(KEY_NOTIFICATION, value).apply()

    // Language
    var language: String
        get() = prefs.getString(KEY_LANGUAGE, "id") ?: "id"
        set(value) = prefs.edit().putString(KEY_LANGUAGE, value).apply()
}

App-Specific Storage

Pengenalan App-Specific Storage

App-Specific Storage adalah area penyimpanan file yang hanya bisa diakses oleh aplikasi Anda. File-file ini otomatis dihapus saat aplikasi di-uninstall.

Jenis App-Specific Storage

Jenis Lokasi Perlu Permission? Keterangan
Internal Storage /data/data/package/files/ Tidak Selalu tersedia, private
Internal Cache /data/data/package/cache/ Tidak Untuk file temporary, bisa dihapus sistem
External App Storage /sdcard/Android/data/package/ Tidak (API 19+) Bisa tidak tersedia jika SD card dicabut

Internal Storage

Internal Storage adalah tempat paling aman untuk menyimpan file karena tidak bisa diakses oleh aplikasi lain atau user (tanpa root).

Mendapatkan Direktori

Mendapatkan path direktori

// Path: /data/data/com.example.app/files/

// Direktori cache internal

Menulis File ke Internal Storage

// Cara 1: Menggunakan openFileOutput (Activity method)
fun saveTextToFile(filename: String, content: String) {
    openFileOutput(filename, Context.MODE_PRIVATE).use { outputStream ->
        outputStream.write(content.toByteArray())
    }
    // File tersimpan di: /data/data/package/files/filename
}

// Cara 2: Menggunakan File object
fun saveTextToFile2(filename: String, content: String) {
    val file = File(filesDir, filename)
    file.writeText(content)
}

// Cara 3: Append (menambahkan ke file yang sudah ada)
fun appendTextToFile(filename: String, content: String) {
    openFileOutput(filename, Context.MODE_APPEND).use { outputStream ->
        outputStream.write(content.toByteArray())
    }
}

// Penggunaan:

appendTextToFile("log.txt", "Log entry: ${System.currentTimeMillis()}\n")

Mode Keterangan
Context.MODE_PRIVATE Membuat file baru atau menimpa file yang sudah ada
Context.MODE_APPEND Menambahkan data ke akhir file yang sudah ada

Buka NotesActivity.kt dan isi dengan kode berikut. Fungsi saveNote() menyimpan isi EditText ke file .txt di filesDir, sedangkan loadNote() membacanya kembali. Nama file dibuat otomatis dari judul catatan.

NotesActivity.kt
package com.example.notesapp

import android.os.Bundle
import android.widget.Button
import android.widget.EditText
import android.widget.TextView
import android.widget.Toast
import androidx.appcompat.app.AppCompatActivity
import java.io.File

class NotesActivity : AppCompatActivity() {

    private lateinit var etTitle: EditText
    private lateinit var etContent: EditText
    private lateinit var tvStatus: TextView

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_notes)

        etTitle = findViewById(R.id.etTitle)
        etContent = findViewById(R.id.etContent)
        tvStatus = findViewById(R.id.tvStatus)

        findViewById<Button>(R.id.btnSave).setOnClickListener { saveNote() }
        findViewById<Button>(R.id.btnLoad).setOnClickListener { loadNote() }
    }

    private fun saveNote() {
        val title = etTitle.text.toString().trim()
        val content = etContent.text.toString().trim()

        if (title.isEmpty()) {
            etTitle.error = "Judul tidak boleh kosong"
            return
        }

        try {
            // Nama file dibuat dari judul, spasi diganti underscore
            // Contoh judul "Belajar Kotlin" → file "belajar_kotlin.txt"
            val filename = "${title.replace(" ", "_").lowercase()}.txt"

            // filesDir → /data/data/com.example.notesapp/files/
            val file = File(filesDir, filename)
            file.writeText(content)

            // Tampilkan path lengkap agar mahasiswa tahu di mana file disimpan
            tvStatus.text = "Tersimpan: ${file.absolutePath}"
            Toast.makeText(this, "Catatan berhasil disimpan!", Toast.LENGTH_SHORT).show()
        } catch (e: Exception) {
            Toast.makeText(this, "Gagal menyimpan: ${e.message}", Toast.LENGTH_SHORT).show()
        }
    }

    private fun loadNote() {
        val title = etTitle.text.toString().trim()

        if (title.isEmpty()) {
            etTitle.error = "Masukkan judul catatan yang akan dimuat"
            return
        }

        try {
            val filename = "${title.replace(" ", "_").lowercase()}.txt"
            val file = File(filesDir, filename)

            if (file.exists()) {
                val content = file.readText()
                etContent.setText(content)
                tvStatus.text = "Dimuat: $filename (${file.length()} bytes)"
                Toast.makeText(this, "Catatan berhasil dimuat!", Toast.LENGTH_SHORT).show()
            } else {
                Toast.makeText(this, "File tidak ditemukan: $filename", Toast.LENGTH_SHORT).show()
            }
        } catch (e: Exception) {
            Toast.makeText(this, "Gagal memuat: ${e.message}", Toast.LENGTH_SHORT).show()
        }
    }
}

SQLite Database

Pengenalan SQLite

SQLite adalah database relasional ringan yang sudah terintegrasi langsung di dalam Android. Berbeda dengan MySQL atau PostgreSQL yang memerlukan server terpisah, SQLite berjalan sepenuhnya di dalam aplikasi itu sendiri — tidak ada proses server, tidak ada koneksi jaringan, tidak ada konfigurasi tambahan.

Mengapa bukan MySQL?

MySQL dirancang untuk berjalan sebagai server yang melayani banyak klien sekaligus melalui jaringan. Di aplikasi mobile, pendekatan ini tidak praktis karena:

  • Tidak ada server — perangkat Android tidak menjalankan server database. MySQL butuh proses server yang selalu aktif dan mengonsumsi banyak memori.
  • Tidak ada jaringan lokal — MySQL berkomunikasi lewat TCP/IP. Koneksi ke server MySQL eksternal berarti butuh internet, latensi, dan penanganan error jaringan.
  • Overkill untuk data lokal — MySQL cocok untuk data yang diakses banyak pengguna secara bersamaan. Data lokal di satu perangkat tidak memerlukan fitur sebesar itu.

SQLite hadir sebagai solusi: database penuh yang berjalan dalam satu file, tanpa server, tanpa konfigurasi, langsung tersedia di setiap perangkat Android.

Karakteristik SQLite

  • Serverless — tidak memerlukan proses server terpisah, berjalan langsung di dalam aplikasi
  • Zero Configuration — tidak perlu setup, instalasi, atau administrasi apapun
  • Self-contained — seluruh database tersimpan dalam satu file .db
  • ACID Compliant — mendukung transaction untuk menjaga konsistensi data
  • SQL Standard — mendukung sebagian besar sintaks SQL standar

database/PhoneDbHelper.kt
package com.example.phoneapp.database

import android.content.ContentValues
import android.content.Context
import android.database.sqlite.SQLiteDatabase
import android.database.sqlite.SQLiteOpenHelper
import com.example.phoneapp.model.Phone

class PhoneDbHelper(context: Context) : SQLiteOpenHelper(
    context, "PhoneDB", null, 1
) {

    companion object {
        const val TABLE      = "phones"
        const val COL_ID     = "id"
        const val COL_MEREK  = "merek"
        const val COL_PRODUSEN = "produsen"
    }

    // Dipanggil SEKALI saat database pertama kali dibuat
    override fun onCreate(db: SQLiteDatabase?) {
        db?.execSQL("""
            CREATE TABLE $TABLE (
                $COL_ID       INTEGER PRIMARY KEY AUTOINCREMENT,
                $COL_MEREK    TEXT NOT NULL,
                $COL_PRODUSEN TEXT NOT NULL
            )
        """.trimIndent())
    }

    // Dipanggil saat versi database dinaikkan
    override fun onUpgrade(db: SQLiteDatabase?, oldVersion: Int, newVersion: Int) {
        db?.execSQL("DROP TABLE IF EXISTS $TABLE")
        onCreate(db)
    }

    // INSERT — tambah data baru, kembalikan ID baris baru (-1 jika gagal)
    fun insert(phone: Phone): Long {
        val db = writableDatabase
        val values = ContentValues().apply {
            put(COL_MEREK, phone.merek)
            put(COL_PRODUSEN, phone.produsen)
        }
        val id = db.insert(TABLE, null, values)
        db.close()
        return id
    }

    // SELECT ALL — ambil semua data, urutkan berdasarkan merek
    fun getAll(): List<Phone> {
        val list = mutableListOf<Phone>()
        val db = readableDatabase
        val cursor = db.rawQuery("SELECT * FROM $TABLE ORDER BY $COL_MEREK ASC", null)

        if (cursor.moveToFirst()) {
            do {
                list.add(
                    Phone(
                        id       = cursor.getInt(cursor.getColumnIndexOrThrow(COL_ID)),
                        merek    = cursor.getString(cursor.getColumnIndexOrThrow(COL_MEREK)),
                        produsen = cursor.getString(cursor.getColumnIndexOrThrow(COL_PRODUSEN))
                    )
                )
            } while (cursor.moveToNext())
        }

        cursor.close()
        db.close()
        return list
    }

    // UPDATE — perbarui data berdasarkan ID, kembalikan jumlah baris terpengaruh
    fun update(phone: Phone): Int {
        val db = writableDatabase
        val values = ContentValues().apply {
            put(COL_MEREK, phone.merek)
            put(COL_PRODUSEN, phone.produsen)
        }
        val rows = db.update(TABLE, values, "$COL_ID = ?", arrayOf(phone.id.toString()))
        db.close()
        return rows
    }

    // DELETE — hapus data berdasarkan ID, kembalikan jumlah baris terhapus
    fun delete(id: Int): Int {
        val db = writableDatabase
        val rows = db.delete(TABLE, "$COL_ID = ?", arrayOf(id.toString()))
        db.close()
        return rows
    }
}

MainActivity.kt
package com.example.phoneapp

import android.os.Bundle
import android.widget.*
import androidx.appcompat.app.AlertDialog
import androidx.appcompat.app.AppCompatActivity
import com.example.phoneapp.database.PhoneDbHelper
import com.example.phoneapp.model.Phone

class MainActivity : AppCompatActivity() {

    private lateinit var dbHelper: PhoneDbHelper
    private lateinit var listView: ListView
    private lateinit var etMerek: EditText
    private lateinit var etProdusen: EditText
    private var phoneList = listOf<Phone>()

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        dbHelper   = PhoneDbHelper(this)
        listView   = findViewById(R.id.listView)
        etMerek    = findViewById(R.id.etMerek)
        etProdusen = findViewById(R.id.etProdusen)

        refreshList()

        // Tombol Tambah — panggil insert() lalu refresh tampilan
        findViewById<Button>(R.id.btnTambah).setOnClickListener {
            val merek    = etMerek.text.toString().trim()
            val produsen = etProdusen.text.toString().trim()

            if (merek.isEmpty() || produsen.isEmpty()) {
                Toast.makeText(this, "Merek dan produsen harus diisi", Toast.LENGTH_SHORT).show()
                return@setOnClickListener
            }

            val id = dbHelper.insert(Phone(merek = merek, produsen = produsen))
            if (id != -1L) {
                Toast.makeText(this, "Berhasil ditambahkan (ID: $id)", Toast.LENGTH_SHORT).show()
                etMerek.text.clear()
                etProdusen.text.clear()
                refreshList()
            }
        }

        // Long-press item untuk hapus — gunakan ID dari database, bukan posisi list
        listView.setOnItemLongClickListener { _, _, position, _ ->
            val phone = phoneList[position]
            AlertDialog.Builder(this)
                .setTitle("Hapus Data")
                .setMessage("Hapus \"${phone.merek}\" dari daftar?")
                .setPositiveButton("Hapus") { _, _ ->
                    dbHelper.delete(phone.id)
                    refreshList()
                    Toast.makeText(this, "${phone.merek} dihapus", Toast.LENGTH_SHORT).show()
                }
                .setNegativeButton("Batal", null)
                .show()
            true
        }
    }

    // Ambil semua data dari DB dan tampilkan di ListView
    private fun refreshList() {
        phoneList = dbHelper.getAll()
        val items = phoneList.map { "${it.merek}  —  ${it.produsen}" }
        listView.adapter = ArrayAdapter(this, android.R.layout.simple_list_item_1, items)
    }

    override fun onDestroy() {
        dbHelper.close()
        super.onDestroy()
    }
}

Kelemahan SQLite Manual

  • Banyak boilerplate code — ContentValues, Cursor, getColumnIndexOrThrow harus ditulis manual setiap saat
  • Tidak ada validasi query saat compile time — typo pada nama kolom baru ketahuan saat runtime (app crash)
  • Pengelolaan versi dan migration database cukup rumit
  • Tidak thread-safe secara default — operasi database di main thread bisa membekukan UI

Karena kelemahan ini, Google merekomendasikan Room Database sebagai abstraksi di atas SQLite yang mengatasi semua masalah di atas.

Room Database

Pengenalan Room

Room adalah library Jetpack yang menyediakan abstraksi di atas SQLite. Alih-alih menulis ContentValues, Cursor, dan query SQL secara manual seperti di SQLite, Room menggunakan annotation pada Kotlin class sehingga kode jauh lebih ringkas dan aman.

Room tetap menggunakan SQLite di balik layar — bedanya, Room yang mengurus semua boilerplate. Validasi query juga dilakukan saat compile time, bukan saat app berjalan, sehingga typo pada nama kolom langsung ketahuan sebelum app dijalankan.

Perbandingan dengan SQLite Manual

Aspek SQLite Manual Room
Validasi query Runtime — crash saat app jalan Compile time — error saat build
Boilerplate code Banyak (ContentValues, Cursor, getColumnIndex) Minimal — cukup annotation
Mapping ke object Manual, baris per baris dari Cursor Otomatis via @Entity
Threading Harus dikelola sendiri Built-in dengan suspend/coroutines
Migration Manual dan rawan error Terstruktur dan lebih aman

Apa itu AGP (Android Gradle Plugin)?

AGP adalah plugin Gradle yang menjadi "mesin" build aplikasi Android. AGP yang mengurus proses kompilasi kode, packaging APK, menjalankan ProGuard, dan banyak hal lain di balik layar saat kamu klik Run atau Build.

Versi AGP biasanya mengikuti versi Android Studio — semakin baru Android Studio, semakin baru AGP yang di-generate otomatis. Masalahnya, tool lain seperti KAPT dan KSP butuh waktu untuk menyesuaikan diri dengan perubahan AGP baru. AGP 9.x (dirilis Januari 2026) masih terlalu baru dan menyebabkan error saat dipakai bersama Room.

Untuk praktikum ini, kita gunakan AGP 8.7.3 yang sudah stabil dan terbukti kompatibel.

Ganti baris agp di libs.versions.toml dari versi yang di-generate Android Studio menjadi 8.7.3, kemudian ikuti langkah berikutnya untuk menambahkan dependency Room.

@Entity adalah pengganti data class biasa sekaligus definisi tabel. Tidak perlu lagi menulis CREATE TABLE secara manual — Room membacanya dari annotation ini.

database/PhoneEntity.kt
package com.example.phoneroom.database

import androidx.room.ColumnInfo
import androidx.room.Entity
import androidx.room.PrimaryKey

// @Entity memberitahu Room bahwa class ini adalah sebuah tabel di database.
// tableName = nama tabel yang akan dibuat di SQLite (boleh berbeda dari nama class).
@Entity(tableName = "phones")
data class PhoneEntity(

    // @PrimaryKey(autoGenerate = true) = kolom id yang auto increment.
    // Sama seperti "id INTEGER PRIMARY KEY AUTOINCREMENT" di SQLite manual.
    @PrimaryKey(autoGenerate = true)
    val id: Int = 0,

    // @ColumnInfo(name = "...") = nama kolom di tabel.
    // Jika nama property sama dengan nama kolom, annotation ini bisa dihilangkan.
    @ColumnInfo(name = "merek")
    val merek: String,

    @ColumnInfo(name = "produsen")
    val produsen: String
)

// Bandingkan dengan SQLite manual:
// Tidak ada lagi CREATE TABLE, tidak ada COL_ID, COL_MEREK, COL_PRODUSEN.
// Semua sudah terdefinisi di sini lewat annotation.

Buat PhoneDao.kt

Klik kanan package database → New → Kotlin Class/File → pilih Interface → beri nama PhoneDao.

@Dao adalah pengganti semua method insert(), getAll(), update(), delete() yang kita tulis manual di PhoneDbHelper. Di sini kita hanya deklarasikan apa yang ingin dilakukan — Room yang generate implementasinya saat build.

database/PhoneDao.kt
package com.example.phoneroom.database

import androidx.room.Dao
import androidx.room.Delete
import androidx.room.Insert
import androidx.room.OnConflictStrategy
import androidx.room.Query
import androidx.room.Update

// @Dao menandai interface ini sebagai Data Access Object.
// Room akan otomatis generate class implementasinya saat project di-build.
@Dao
interface PhoneDao {

    // @Insert: Room generate query INSERT secara otomatis.
    // onConflict = REPLACE artinya jika ada data dengan id sama, timpa data lama.
    // suspend = harus dipanggil dari coroutine (tidak boleh di main thread).
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insert(phone: PhoneEntity): Long

    // @Query: untuk SELECT, kita tulis SQL-nya sendiri.
    // Room memvalidasi query ini saat compile time — typo nama kolom langsung error saat build.
    @Query("SELECT * FROM phones ORDER BY merek ASC")
    suspend fun getAll(): List<PhoneEntity>

    // @Update: Room generate UPDATE berdasarkan primary key (id) secara otomatis.
    // Mengembalikan jumlah baris yang berhasil diperbarui.
    @Update
    suspend fun update(phone: PhoneEntity): Int

    // @Delete: Room generate DELETE berdasarkan primary key secara otomatis.
    // Mengembalikan jumlah baris yang berhasil dihapus.
    @Delete
    suspend fun delete(phone: PhoneEntity): Int
}

// Bandingkan dengan SQLite manual:
// Tidak ada lagi ContentValues, writableDatabase, readableDatabase,
// Cursor, moveToFirst(), getColumnIndexOrThrow(), cursor.close(), db.close().
// Semua itu digantikan oleh 4 baris annotation di atas.

Buat PhoneDatabase.kt

Klik kanan package database → New → Kotlin Class/File → beri nama PhoneDatabase.

@Database adalah pengganti SQLiteOpenHelper. Class ini menjadi satu-satunya pintu masuk ke database dan dibuat sebagai Singleton agar tidak ada lebih dari satu koneksi database yang terbuka sekaligus.

database/PhoneDatabase.kt
package com.example.phoneroom.database

import android.content.Context
import androidx.room.Database
import androidx.room.Room
import androidx.room.RoomDatabase

// @Database mendaftarkan semua Entity dan menentukan versi database.
// Jika struktur tabel berubah (tambah kolom, dll), naikkan version dan buat Migration.
@Database(
    entities = [PhoneEntity::class],  // daftar semua tabel
    version = 1,
    exportSchema = false
)
abstract class PhoneDatabase : RoomDatabase() {

    // Room akan generate implementasi fungsi ini secara otomatis.
    abstract fun phoneDao(): PhoneDao

    companion object {
        @Volatile
        private var INSTANCE: PhoneDatabase? = null

        // Singleton: pastikan hanya ada satu instance database di seluruh app.
        // @Volatile memastikan perubahan INSTANCE langsung terlihat oleh semua thread.
        fun getInstance(context: Context): PhoneDatabase {
            return INSTANCE ?: synchronized(this) {
                Room.databaseBuilder(
                    context.applicationContext,
                    PhoneDatabase::class.java,
                    "phone_database"  // nama file .db yang tersimpan di /data/data/package/databases/
                )
                .fallbackToDestructiveMigration()  // untuk development: hapus & buat ulang jika versi naik
                .build()
                .also { INSTANCE = it }
            }
        }
    }
}

Implementasi MainActivity.kt

Perbedaan utama dari versi SQLite: semua operasi database dipanggil di dalam lifecycleScope.launch karena Room menggunakan suspend function. Tidak ada lagi dbHelper.close() — Room mengelola koneksi secara otomatis.

MainActivity.kt
package com.example.phoneroom

import android.os.Bundle
import android.widget.*
import androidx.appcompat.app.AlertDialog
import androidx.appcompat.app.AppCompatActivity
import androidx.lifecycle.lifecycleScope
import com.example.phoneroom.database.PhoneDatabase
import com.example.phoneroom.database.PhoneEntity
import kotlinx.coroutines.launch

class MainActivity : AppCompatActivity() {

    // Akses database lewat DAO, bukan lewat helper class seperti SQLite manual
    private lateinit var dao: com.example.phoneroom.database.PhoneDao
    private lateinit var listView: ListView
    private lateinit var etMerek: EditText
    private lateinit var etProdusen: EditText
    private var phoneList = listOf<PhoneEntity>()

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // Inisialisasi database — Room membuat file .db secara otomatis jika belum ada
        dao        = PhoneDatabase.getInstance(this).phoneDao()
        listView   = findViewById(R.id.listView)
        etMerek    = findViewById(R.id.etMerek)
        etProdusen = findViewById(R.id.etProdusen)

        refreshList()

        // Tombol Tambah
        // lifecycleScope.launch diperlukan karena dao.insert() adalah suspend function
        findViewById<Button>(R.id.btnTambah).setOnClickListener {
            val merek    = etMerek.text.toString().trim()
            val produsen = etProdusen.text.toString().trim()

            if (merek.isEmpty() || produsen.isEmpty()) {
                Toast.makeText(this, "Merek dan produsen harus diisi", Toast.LENGTH_SHORT).show()
                return@setOnClickListener
            }

            lifecycleScope.launch {
                // Tidak perlu ContentValues, cukup buat object PhoneEntity
                val id = dao.insert(PhoneEntity(merek = merek, produsen = produsen))
                Toast.makeText(this@MainActivity, "Ditambahkan (ID: $id)", Toast.LENGTH_SHORT).show()
                etMerek.text.clear()
                etProdusen.text.clear()
                refreshList()
            }
        }

        // Long-press untuk hapus
        listView.setOnItemLongClickListener { _, _, position, _ ->
            val phone = phoneList[position]
            AlertDialog.Builder(this)
                .setTitle("Hapus Data")
                .setMessage("Hapus \"${phone.merek}\" dari daftar?")
                .setPositiveButton("Hapus") { dialog, _ ->
                    lifecycleScope.launch {
                        dao.delete(phone)  // cukup kirim object, Room tahu ID-nya
                        refreshList()
                        Toast.makeText(this@MainActivity, "${phone.merek} dihapus", Toast.LENGTH_SHORT).show()
                    }
                }
                .setNegativeButton("Batal", null)
                .show()
            true
        }
    }

    private fun refreshList() {
        // lifecycleScope: coroutine yang otomatis batal saat Activity dihancurkan
        lifecycleScope.launch {
            phoneList = dao.getAll()
            val items = phoneList.map { "${it.merek}  —  ${it.produsen}" }
            listView.adapter = ArrayAdapter(this@MainActivity, android.R.layout.simple_list_item_1, items)
        }
    }

    // Tidak perlu override onDestroy() untuk menutup database
    // Room mengelola koneksi secara otomatis
}

Di mana database Room disimpan?

Room tetap menyimpan database di lokasi yang sama dengan SQLite manual:

Nama file sesuai dengan string yang ditulis di Room.databaseBuilder(). Room pada dasarnya adalah SQLite — hanya lapisan abstraksinya yang berbeda. Di balik layar Room tetap membuat file .db di folder databases/ yang sama. Untuk memverifikasi, buka Device File Explorer di Android Studio → navigasi ke path tersebut.
